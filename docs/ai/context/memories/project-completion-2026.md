# Project Completion Rules (2026 PRIORITY #1)

Last Updated: 2026-01-06

## PRIMARY GOAL

Complete projects to production-ready state using up-to-date information

## Core Principles

1. **Finish What You Start**
   - Complete features to 100% before moving to next task
   - Don't leave half-implemented features
   - Mark complete ONLY when fully working

2. **Use Current Information (2026)**
   - Always verify libraries/tools are current
   - Use web search to validate latest versions
   - Check official documentation for breaking changes
   - Don't assume 2024/2025 knowledge is valid

3. **Production-Ready Quality**
   - All features must be tested
   - All features must be documented
   - All features must be integrated and working
   - No "TODO" or "FIXME" in completed work

4. **Verification Before Completion**
   - Test each feature after implementation
   - Verify integration with existing codebase
   - Check dependencies are up-to-date
   - Confirm no regressions introduced

## Completion Checklist

Before marking ANY task as complete:

- [ ] Feature is fully implemented (no placeholders)
- [ ] All code is tested (unit/integration/E2E)
- [ ] Documentation is complete and accurate
- [ ] Dependencies are latest stable versions (2026)
- [ ] Integration with existing features verified
- [ ] No console errors or warnings
- [ ] Performance is acceptable
- [ ] Code follows project standards
- [ ] Git commit is clean and descriptive
- [ ] User can actually use the feature (end-to-end)

## Anti-Patterns (NEVER DO THIS)

1. Half-Finished Features
   - ❌ "This mostly works" - NO, must fully work
   - ❌ "We can finish later" - NO, finish now
   - ❌ "Just need tests" - NO, tests are part of completion

2. Outdated Information
   - ❌ "React 18 works this way" - NO, verify React 19 docs
   - ❌ "This worked in 2024" - NO, check 2026 practices
   - ❌ "I remember this API" - NO, check current docs

3. Incomplete Integration
   - ❌ "Component exists" - NO, must be integrated and working
   - ❌ "Service is written" - NO, must be connected and tested
   - ❌ "Feature is coded" - NO, users must be able to use it

4. Skipping Verification
   - ❌ "Looks good" - NO, test it
   - ❌ "Should work" - NO, verify it works
   - ❌ "Probably correct" - NO, confirm it's correct

## When to Mark Complete

Task is complete ONLY when ALL are true:

1. ✅ Code is written and working
2. ✅ Tests are passing
3. ✅ Documentation is written
4. ✅ Feature is integrated into main app
5. ✅ User can access and use the feature
6. ✅ No regressions introduced
7. ✅ Dependencies are up-to-date (2026)
8. ✅ Code follows project standards
9. ✅ Performance is acceptable
10. ✅ YOU would ship this to production

If ANY is false, task is NOT complete.

## Remember

A completed project that works is infinitely better than 10 half-finished projects.
